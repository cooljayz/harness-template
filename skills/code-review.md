# Skill: code-review

**When to use**: 에이전트가 자신 또는 다른 에이전트가 쓴 변경을 리뷰할 때. LLM 친화적 의미 리뷰.

## 리뷰 순서 (우선순위대로)

### 1. 아키텍처 (Structural)
- [ ] `docs/architecture.md`의 계층 규칙을 위반하지 않는가
- [ ] 순환 의존이 생기지 않는가
- [ ] 외부 I/O가 Repo 계층 밖으로 새지 않는가
- [ ] 구조 테스트가 통과하는가

### 2. 설계 정합성 (Intent)
- [ ] 변경이 `docs/design/`의 해당 설계 문서와 일치하는가
- [ ] 설계 문서에 없는 추가 기능("그 김에")이 섞이지 않았는가
- [ ] 분해 단위(depth-first 블록)가 지켜졌는가

### 3. 센서 (Feedback)
- [ ] 관측성 표준(`docs/quality/observability.md`)을 따르는가
- [ ] 실패 신호가 에이전트가 읽을 수 있는 형태인가
- [ ] 등급에 맞는 테스트가 있는가

### 4. 문서 동기화
- [ ] 동작이 바뀌었다면 관련 문서도 같은 PR에서 갱신되었는가
- [ ] 플랜 문서(`docs/plans/`)의 체크리스트가 갱신되었는가

### 5. Karpathy 체크 (행동 원칙 준수)
- [ ] **Simplicity**: 요청하지 않은 기능/추상화/유연성이 추가되지 않았는가
- [ ] **Surgical**: 변경된 모든 줄이 사용자 요청으로 직접 추적 가능한가
- [ ] **Surgical**: 기존 인접 코드·주석·포매팅을 불필요하게 건드리지 않았는가
- [ ] **Surgical**: 내 변경으로 생긴 고아(unused import 등)는 정리했는가
- [ ] **Goal-Driven**: 성공 기준(테스트/검증)이 있고 그것이 통과하는가

### 6. 의미 (Semantic — LLM의 강점)
- [ ] 이름이 의도를 드러내는가
- [ ] 예외 처리가 실패 모드를 숨기지 않는가
- [ ] 데드 코드/중복이 없는가

## 출력 형식

리뷰는 다음 구조의 마크다운으로 남긴다:

```
## Blocking
- <반드시 고쳐야 하는 것>

## Suggest
- <권장>

## Nit
- <사소한 것>
```
