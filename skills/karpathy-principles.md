# Skill: karpathy-principles

Andrej Karpathy의 LLM 코딩 실수 관찰에서 도출된 행동 지침. 모든 코딩 작업의 기본값으로 적용한다.
`AGENTS.md` 섹션 4의 상세 버전.

**트레이드오프**: 이 지침은 속도보다 신중함을 선택한다. 사소한 작업은 판단해서 조정한다.

---

## 1. Think Before Coding — 가정하지 말고 표면화하라

구현 전에:
- 가정을 명시한다. 불확실하면 묻는다.
- 여러 해석이 있으면 조용히 고르지 말고 제시한다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반론한다.
- 무언가 불분명하면 멈추고, 무엇이 불분명한지 이름 붙여 질문한다.

**Harness 연결**: `docs/design/`의 설계 문서가 없으면 코드 전에 설계 문서를 먼저 작성한다.

## 2. Simplicity First — 요청한 것만, 최소한으로

- 요청하지 않은 기능 추가 금지.
- 단일 사용 코드에 추상화 레이어 금지.
- 요청하지 않은 "유연성"이나 "확장성" 금지.
- 불가능한 시나리오의 오류 처리 금지.
- 200줄짜리가 50줄로 될 수 있으면 다시 쓴다.

스스로 질문: "시니어 엔지니어가 이걸 보고 과하다고 할까?" → 그렇다면 단순화.

**Harness 연결**: depth-first 블록 분해와 맞닿는다. 한 블록에서 다음 블록 범위를 침범하지 않는다.

## 3. Surgical Changes — 필요한 것만 건드려라

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포매팅을 "개선"하지 않는다.
- 고장나지 않은 것을 리팩토링하지 않는다.
- 기존 스타일을 맞춘다, 내가 다르게 하더라도.
- 관련 없는 데드 코드를 발견하면 **언급하되 삭제하지 않는다**.

내 변경이 고아를 만들 때:
- **내 변경**으로 쓸모없어진 import/변수/함수는 제거한다.
- 기존에 있던 데드 코드는 요청 없이 건드리지 않는다.

기준: 변경된 모든 줄이 사용자 요청으로 직접 추적 가능해야 한다.

**Harness 연결**: 문서 갱신도 같은 원칙 — 내 변경으로 낡아진 문서만 갱신, 전혀 다른 섹션은 건드리지 않는다.

## 4. Goal-Driven Execution — 성공 기준을 정의하고 루프

작업을 검증 가능한 목표로 변환한다:
- "검증 추가" → "잘못된 입력에 대한 테스트 작성, 그 테스트를 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성, 그 테스트를 통과시키기"
- "리팩토링" → "수정 전후 테스트가 통과하는지 확인"

다단계 작업에서는 간략한 계획을 먼저 제시:
```
1. [단계] → 검증: [확인 항목]
2. [단계] → 검증: [확인 항목]
3. [단계] → 검증: [확인 항목]
```

강한 성공 기준 → 독립적으로 루프 가능.
약한 기준("되게 해줘") → 계속 확인 요청 필요.

**Harness 연결**: `scripts/test-fast.sh`, `tests/architecture/`가 자동 검증 루프의 센서다.
에이전트는 센서가 통과할 때까지 루프를 닫지 않는다.

## 이 원칙이 잘 작동하고 있다면

- 불필요한 변경이 diff에 줄어든다
- 과도한 구현으로 인한 재작성이 줄어든다
- 구현 전에 명확화 질문이 온다 (구현 후 실수가 아니라)
